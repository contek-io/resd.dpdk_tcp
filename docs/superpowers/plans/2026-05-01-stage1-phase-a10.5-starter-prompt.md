## A10.5 scope

From the roadmap entry at `docs/superpowers/plans/stage1-phase-roadmap.md`
under "## A10.5 — Layer H correctness under WAN-condition fault injection":

**Goal**: Promote spec §10.10's informal "end-to-end smoke under tc netem
loss/delay" to a named Layer H test phase. Formal netem matrix with
**liveness + invariant assertions**, not performance measurement. A10
measures *how fast* the stack runs under adversity; A10.5 asserts *that
the stack stays correct* under the same adversity.

**Deliverables (from the roadmap)**:
- `tools/layer-h-correctness/` — netem matrix runner producing pass/fail
  per scenario, not p50/p99/p999. Reuses A10's `bench-stress` netem
  scaffolding (driver, precondition checker, scenario harness) as a
  library; only the assertion layer is new.
- Netem matrix (Stage 1 subset of §10.8):
  - Delay: +20 ms, +50 ms, +200 ms (each with and without jitter)
  - Loss: 0.1%, 1%, 5% random; 1% correlated bursts
  - Duplication: 0.5%, 2%
  - Reordering: depth 3
  - Corruption: 0.01% (checksum-fail drops expected; asserted via
    `rx_drop_cksum_bad`)
- Composable with A9's FaultInjector env-var side channel so RX-side and
  WAN-side adversity can be stacked.
- Assertion table per scenario:
  - Connection stays ESTABLISHED for configured duration — no unexpected
    Closed/Error transition
  - `tcp.tx_retrans` bounded by `max_retrans_count`; no retransmit storms
  - FSM state ∈ legal set per §6.1 throughout
  - `obs.events_dropped == 0` at steady load
  - Per-scenario expected counter-signals (e.g. 1% loss → nonzero
    `tcp.tx_retrans`; reorder-3 → nonzero `rx_dup_ack` without crossing
    3-dup-ACK fast-retransmit unless preset permits)
- CI: Layer-H smoke (one representative bucket per dimension) per merge;
  full matrix per stage cut, report to
  `docs/superpowers/reports/layer-h-<date>.md`.
- End-of-phase mTCP + RFC review gates.

**Does NOT include**:
- PMTU-blackholing (Stage 2 — needs PLPMTUD)
- Performance / latency under netem (A10's bench-stress owns that)
- Layer G WAN A/B vs Linux (Stage 2-A)
- New counters or events (asserts against existing observability surface)
- Fuzzing / proptest (A9 territory)

**Dependencies**: A10 (netem harness, scaffolding), A9 (FaultInjector).
Blocks A11.

**Rough scale**: ~6–8 tasks.
